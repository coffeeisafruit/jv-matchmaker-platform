Step 4: Exa Research PAID — neural search

Final Score Formula (Two-Level)

Directional: S = exp(Σwi·log(xi) / Σwi)
Pair: H = 2 / (1/Sab + 1/Sba)
Level 1: Weighted geometric mean across ISMC dimensions per direction (0-100).
Level 2: Harmonic mean of directional scores for balanced pair scoring.
Calibrated for production: mean ~57.5, stdev ~5.7, range 25-78.

Tier Thresholds (PartnershipAnalyzer)

Premier: ≥64 (highest confidence, both sides benefit strongly)
Strong: ≥55 (high-confidence, clear mutual value)
Aligned: <55 (compatible partners with developing match signal)

Embedding Calibration (bge-large-en-v1.5)

Cosine similarity ≥0.75 → 10.0 (synonym-level match)
≥0.65 → 8.0  |  ≥0.60 → 6.0  |  ≥0.53 → 4.5 (noise floor)
<0.53 → 3.0 (no signal). Falls back to Jaccard word overlap when embeddings are null.

Match Reason System (5-Layer Signal Analysis)

Human-readable match reasons replace keyword arrays. Generated in _generate_match_reason() and persisted to the match_reason DB field.

Layer 1: Audience Overlap — Extracts shared keywords from who_you_serve fields, filtering stop words and partner name tokens. Detects audience bridges for cross-promotion.
Layer 2: Offering↔Seeking Alignment — Cross-matches seeking keywords against offering items. Identifies direct partnership value alignment.
Layer 3: Collaboration Format — Recognizes concrete next steps from content_platforms (podcast guest swaps, co-hosted webinars, joint programs, email list swaps).
Layer 4: Scale Context — List-size-aware suggestions. Combined audience metrics for co-promotion recommendations when both partners have 5K+ lists.
Layer 5: Niche Fit — Niche alignment scoring with blocklist filtering. Only adds niche context when richer signals are absent.

All output runs through TextSanitizer.validate_match_reason() to strip keyword arrays, warning emojis, and score references.

Key Orchestration Patterns

Calendar-Driven Phases — monthly-orchestrator auto-detects phase from date.today() using ISO week calculations. Falls back to no-op on unscheduled dates.

Hierarchical Task Mapping — Parent flows use .map() to fan out parallel work across profile batches, with configurable concurrency parameter.

Batch Execution with SAVEPOINT — DB writes run inside SAVEPOINTs. If one profile fails, the batch continues. Failed profiles are quarantined for retry.

Cost Attribution per Profile — Every API call logs tool, cost, and profile_id to costs_YYYY-MM.jsonl. Monthly reports aggregate per-tool spending and cost-per-useful-result.

Source Priority Protection — _should_write_field() enforces priority hierarchy: client_confirmed (100) > client_ingest (90) > manual_edit (80) > exa_research (50) > ai_research (40) > apollo (30). Higher-priority data is immutable.

Dry-Run Threading — All flows and tasks accept dry_run: bool. When enabled, full logging runs but no DB writes or external API calls occur.

Frequently Asked Questions

Common questions about how the JV Matchmaker Platform works, keeps data fresh, and scales.

How does the platform keep 1M+ contacts accurate over time?

The platform runs automated monthly refresh cycles orchestrated by Prefect. Every month, a calendar-driven orchestrator dispatches five phases:

  1. Week 1 — Change detection scans all profiles for data drift using content hashes
  2. Week 3 — Client verification emails confirm key profile fields with signed-up members
  3. Week 4 — Full re-enrichment pipeline flags stale (>29 days) and low-confidence (<0.5) profiles, re-researches them via AI, and rescores all matches
  4. Week 4 Tue — Admin notification delivers an AI-generated digest of changes
  5. 1st of month — Fresh reports with new access codes are delivered to members

Each field has its own exponential confidence decay rate — fast-changing fields like “seeking” decay in 30 days, while stable fields like “name” decay over 365 days. This means the system surgically identifies which fields are stale, not just which profiles.

Are signed-up members prioritized for data freshness?

Yes. The re-enrichment queue sorts members first. When the monthly processing flow flags profiles for refresh, it uses a two-level sort:

  1. Member status — Profiles with status = 'Member' are placed at the front of the queue
  2. Confidence score — Within each group, lowest-confidence profiles are enriched first

This ensures that paying members always get freshened data before prospects or non-member resources, even when the system is processing thousands of stale profiles.

What does the enrichment cost at scale?

At current scale (92K+ enriched, 1.17M total profiles), enrichment costs are low because:

  • Only profiles whose confidence has decayed below threshold are re-enriched — not all profiles every cycle
  • The 29-day stale window means roughly 1/4 of profiles need refresh per month
  • AI research uses Claude via OpenRouter at competitive per-token pricing
  • Each run tracks cost via MonthlyProcessingResult.total_cost

At full scale (800K+ contacts), costs are TBD. The v2.0/docs/planning/VAST_AI_INTEGRATION.md plan outlines a path to 90%+ cost reduction using self-hosted open-source models on Vast.ai when monthly AI costs exceed $150.

How does the ISMC scoring system work?

ISMC stands for Ideal Strategic Match Criteria. It evaluates JV partner compatibility across 25 weighted factors grouped into categories:

  • Audience alignment — list size, niche overlap, audience engagement
  • Business fit — revenue tier, offering complementarity, what they do
  • Partnership readiness — JV history, content platforms, seeking/offering match
  • Strategic value — network centrality, brand authority, growth trajectory

Each factor is scored 0–100, then weighted and aggregated. Matches scoring above the tier threshold (currently 64) are surfaced in member reports. The scoring engine runs after every enrichment cycle to keep recommendations current.

Where does the partner data come from?

The platform combines 35+ automated scrapers with AI enrichment:

  • Public registries — BBB (5.4M businesses), Clutch (373K agencies), IRS business leagues (128K), SAM.gov awards
  • Review platforms — Trustpilot (200K+ with real emails/phones), G2, Capterra
  • Government APIs — USAspending (18.2M recipients), FDIC, EPA ECHO, FDA, Grants.gov, SEC EDGAR
  • Tech ecosystems — Shopify Partners, HubSpot Partners, Salesforce AppExchange, AWS Marketplace
  • VC portfolios — Y Combinator, a16z, Sequoia, Techstars, and more

After collection, contacts go through merge/dedup (reducing 908K raw to 828K unique), then AI enrichment adds business context, validates data, and computes confidence scores.

How does the source priority system prevent bad data from overwriting good data?

Every data field carries a source tag and priority level:

client_confirmedPriority 100Member verified their own data
client_ingestPriority 90Data from member sign-up form
manual_editPriority 80Admin manual correction
exa_researchPriority 50Exa web search results
ai_researchPriority 40Claude AI extraction
apolloPriority 30Apollo.io enrichment

During re-enrichment, a lower-priority source cannot overwrite a higher-priority value. This means if a member confirmed their email, no scraper or AI process can replace it.

What happens when retirements, closures, or business changes occur?

Multiple mechanisms catch these changes:

  • Content hash detection (Week 1) — compares current profile data against stored hashes to flag any drift
  • Confidence decay — unverified data naturally loses confidence over time, triggering re-research
  • AI re-enrichment (Week 4) — when a profile is re-researched, the AI pipeline checks the company’s current website, LinkedIn, and public records. If a business is closed, the research will surface that
  • Client verification (Week 3) — signed-up members receive verification emails to confirm or update their own data
  • Inactive exclusion — profiles marked status='Inactive' are excluded from enrichment queues and match recommendations

The platform uses Prefect 3.x for workflow orchestration with 5 scheduled deployments defined in prefect.yaml:

Can the system scale to enrich all 800K+ contacts?

The current architecture handles 1.17M profiles with Vast.ai for bulk enrichment (~$2/18K profiles). For full-scale enrichment of all 1.17M contacts:

  • Tiered selection ensures the highest-value profiles are enriched first (members → high list-size prospects → others)
  • Cost trigger — when monthly AI costs exceed $150 or volume exceeds 500 profiles/month, the platform can switch to self-hosted models via Vast.ai for 90%+ cost reduction
  • Batch processing — the enrichment flow processes profiles in configurable batch sizes with rate limiting
  • Gap-driven sourcing — rather than enriching everything, the system identifies which client gaps need new partners and targets sourcing accordingly
What anti-hallucination measures are in place for AI-enriched data?

Multiple layers prevent AI hallucination:

  • Source citations required — every AI-extracted data point must include its source URL or reference
  • Cross-validation — data confirmed by 2+ independent sources gets a confidence boost (+0.10 for 2 sources, +0.20 for 3+)
  • AI verification step — a separate verification task checks AI research output against the original source material
  • Source-based confidence — AI inference starts at 0.50 base confidence vs. 0.95 for Apollo-verified data
  • Human override — client-confirmed and manual-edit data always takes priority over any AI source
What is the standard client profile template?

Every client profile must meet a production standard before going live. The gold standard reference is the Next Level DeFi profile. Required sections:

  • Client Card — Real person name (2+ words), company, title
  • The Offer — Program name (not just the person's name), program focus (not generic "Business Growth"), target audience (specific), network reach (real number)
  • About Section — Minimum 2 sentences of real bio content, not auto-generated placeholders
  • Credentials — At least 2 items, no truncated text
  • Seeking Goals — At least 1 JV partnership goal
  • Contact Info — Email or booking link must exist

Recommended sections that improve the production score:

  • Partnership Metrics — Commission %, EPC, value/lead, refund rate
  • Pricing Tiers — 2-3 tiers with descriptions
  • Key Positioning Message — Headline + 3 supporting points
  • What Partners Receive — 3+ deliverables
  • Why This Converts — 2+ data-backed conversion reasons
  • Proven Results — At least 1 case study with metrics
  • Partner FAQs — 3+ Q&A pairs
  • Resources & Proof — 2+ links (website, testimonials, replays)

What is the standard partner card format?

Every partner card on the outreach page must meet quality standards based on the Eli Natoli reference card. Each outreach page requires exactly 10 partner contacts.

Required per card:

  • Name — Real person name
  • Company/Tagline — Not empty
  • Audience — Subscriber count + audience description + offering list (e.g., "7K subscribers. Purpose-driven entrepreneurs... Offering: Content Marketing, Course Creation Training...")
  • Why Fit — Must follow 3-part structure: (1) shared audience/connection, (2) alignment with client's goals, (3) suggested next step (e.g., "Explore a joint program building on Marketing Your Truth™")
  • Contact Info — At least 1 of: email, phone, or LinkedIn
  • Detail Note — Italic bio/context at bottom of card

Recommended per card:

  • 2-4 relevant tags (e.g., MARKETING, STRATEGY, 7K LIST)
  • List size badge (top-right number)
  • Website URL
  • Booking/schedule link
  • All 3 contact methods (email + phone + LinkedIn)
What is the standard outreach email format?

Outreach email templates are generated per client (not per partner) using AI. They reference the client's actual offering, audience, and value proposition — not generic boilerplate.

Initial Outreach (required elements):

  • Greeting with {{partner_first_name}} token
  • Opening that references what the client specifically offers (not "I love what you're doing")
  • Value proposition — what the partner gets from this partnership
  • Social proof — 1 credential, metric, or result
  • Soft CTA with real booking link (if available)
  • Signature (name + company)
  • Length: 120–250 words

Follow-Up (required elements):

  • Brief value-add (not just "following up")
  • Restates the value in different words
  • Soft close
  • Length: 50–100 words

The verification agent checks that templates don't contain unfilled placeholders ([their audience], [Insert booking link]) and aren't identical to the known generic boilerplate. Templates are stored in MemberReport.outreach_templates as JSON.

What does "rescoring" mean, and when does it happen?

Rescoring recalculates scores using the latest data. There are two distinct rescoring operations that work at different levels:

1. Profile Rescoring (JV Readiness) — jv_triage.py

Recalculates the jv_readiness_score (0–100) for individual profiles. This is a weighted formula with five components:

Source JV Affinity35%How JV-relevant is the data source? (MunchEye = 100, BBB = 30)
JV Signals25%JV history, affiliate keywords, commission mentions + seeking (AI-inferred intent, +10) + seniority (Apollo: owner/founder +15, vp/director +8) + intent_signal (Apollo, +10)
Audience Signals20%Content platforms, list size, audience tags + who_you_serve (+8), audience_engagement_score (+12/+6)
Business Substance12%Company, website, revenue tier, niche, bio + what_you_do (+15), offering (+12), signature_programs (+10)
Contactability8%Email, phone, LinkedIn, website + booking_link (+20), network_role (+10) + email_confidence (Apollo: ≥0.8 +10, ≥0.5 +5)

Confidence tiers (updated March 2026): Sub-signals use a Fit / Computed / Intent / Apollo framework. Observable facts (what_you_do, offering, signature_programs, booking_link) receive full points. Computed metrics (audience_engagement_score, network_role) receive full points. AI-inferred intent fields (seeking, who_you_serve) receive ~67% points since data is publicly sourced, not self-reported. Apollo-derived fields (seniority +15/+8, intent_signal +10, email_confidence +10/+5) receive full points as they come from verified API data. When profiles are claimed by users, Intent fields can be upgraded to full points as a “verified” boost.

After enrichment fills in fields like what_you_do, offering, seeking, and signature_programs, a profile’s score can jump significantly — for example, from 29 (source + basic contact only) to 55+ (fully enriched with business context and partnership signals).

2. Match Rescoring (ISMC Pair Scoring) — rescore_matches

Recalculates how well pairs of profiles would work as JV partners using the 25-factor ISMC engine. This produces a harmonic mean score that determines the match tier:

  • Premier (64+) — Highest confidence, both sides benefit strongly
  • Strong (55–63) — High-confidence, clear mutual value
  • Aligned (<55) — Compatible partners, developing match signal

Before promoting a new scoring formula, shadow scoring runs both the production and experimental scorers on the same match pairs and compares outputs. This A/B comparison checks Spearman rank correlation, tier reclassification rates, and mean score shift to ensure changes are safe and predictable.

When does rescoring happen?

  1. After batch enrichment — profile rescoring updates readiness scores to reflect newly filled fields
  2. After scoring formula changes — match rescoring recomputes all pair scores with the updated weights
  3. Monthly maintenance cycle — both run as part of the automated pipeline to keep recommendations current
